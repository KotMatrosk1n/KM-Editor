# Documentation Boundary

Status: Priority 0 direction.

KM Editor should keep public documentation useful, accurate, and focused on the project users can see and run.

## Public Documentation

Public repository documentation can include:

- What KM Editor is.
- Supported games and project scope.
- User-facing workflows.
- Safety rules.
- Build, test, and contribution instructions.
- License information.
- Architecture decisions once they are stable enough to guide contributors.
- Public roadmap items that do not depend on private local data or unpublished workflow details.

## Private Or Local Planning

Private or local-only planning should include:

- Local machine paths.
- Private test data and fixtures.
- Unpublished project notes.
- Scratch research notes.
- Temporary migration notes.
- Sensitive workflow context that is not needed to use, build, or contribute to the public project.

## Rule Of Thumb

Public docs should explain what KM Editor does and how people should work with it.

Private notes should explain local context, unfinished thinking, and project-specific details that are not needed by public users or contributors.

When in doubt, keep sensitive planning local until it has been turned into a general product, architecture, or contributor guideline.
